# React Template Starter тАФ Design Spec

- **Date:** 2026-06-08
- **Status:** Draft, pending implementation
- **Location in repo:** `templates/react-starter/`
- **Distribution:** `pnpm dlx degit Innei/Pastel/templates/react-starter my-app`

## 1. Goal

Add a minimal, opinionated React starter to the Pastel monorepo that downstream users can degit into a new project. The starter:

- Builds on **Vite 8** with **React 19** and the stable **React Compiler**.
- Ships **code-inspector-plugin** for Alt-click jump-to-source.
- Generates routes from the file system with **`vite-plugin-route-builder`**.
- Comes pre-themed with the **Pastel** color system (TailwindCSS v4 + `data-theme` dark mode).
- Uses **jotai** for state and **next-themes** for light/dark switching.
- Stays small enough to read top-to-bottom in one sitting.

A single sample app (Home + About + theme toggle) demonstrates the wiring; everything else is left to consumers.

## 2. Non-goals

- No test framework. Vitest is intentionally omitted; consumers add it themselves if needed.
- No state library beyond jotai. Provider is wired up; no demo atoms.
- No router library beyond `react-router` v7 (used internally by `vite-plugin-route-builder`).
- No icon library. The single icon needed (theme toggle) is inlined as SVG.
- No CI configuration, no Storybook, no Husky, no commitlint, no semantic-release.
- The starter is **not** a workspace member of the Pastel monorepo.

## 3. Approach

**Standalone subdirectory inside the monorepo, outside the pnpm workspace.**

The current `pnpm-workspace.yaml` lists only `packages/*` and `docs`, so `templates/*` is naturally excluded тАФ no changes to workspace config are required.

The starter's `package.json` references published Pastel packages with real npm version specifiers (`@pastel-palette/colors@^1.0.0-canary.3`, `@pastel-palette/tailwindcss@^1.0.0-canary.3`), **not** `workspace:*`. This means:

- `pnpm dlx degit Innei/Pastel/templates/react-starter my-app && cd my-app && pnpm i` works as-is for external users.
- Local development of the starter installs Pastel from npm. Iterating on Pastel itself and testing changes in the starter requires `pnpm pack` / `yalc` ad-hoc; this is acceptable because the starter's primary audience is external consumers.

Alternatives considered and rejected:

- **Workspace member + degit-ready snapshot branch:** would let the starter use `workspace:*` and auto-sync with Pastel changes, but adds a publish/branch machinery for negligible benefit.
- **Hybrid (workspace + postinstall rewrite):** high complexity, fragile degit behavior.

### Conventions

- **`components/ui/`** тАФ primitives with no business knowledge (button, input, etc.).
- **`features/<domain>/`** тАФ a self-contained domain folder. Permitted children: `components/`, `atoms.ts`, `hooks/`, `index.ts` (barrel). Pages import features; features do not import pages.
- **`pages/`** тАФ pure route shells. Each file exports a named `Component` (and optional `loader`) per the `vite-plugin-route-builder` contract. Pages compose features.
- **`lib/`** тАФ pure cross-feature utilities. No React, no business state.

## 6. Tooling configuration

### `vite.config.ts`

```ts

import tailwindcss from '@tailwindcss/vite'

import { codeInspectorPlugin } from 'code-inspector-plugin'
import { defineConfig } from 'vite'
import { routeBuilderPlugin } from 'vite-plugin-route-builder'

export default defineConfig({
  resolve: {
    alias: {
      '~': fileURLToPath(new URL('./src', import.meta.url)),
    },
  },
  plugins: [
    codeInspectorPlugin({
      bundler: 'vite',
      editor: 'cursor',
      hotKeys: ['altKey'],
    }),
    react(),
    babel({ presets: [reactCompilerPreset()] }),
    tailwindcss(),
    routeBuilderPlugin({
      pagePattern: './src/pages/**/*.{tsx,sync.tsx}',
      outputPath: './src/generated-routes.ts',
      enableInDev: true,
    }),
  ],
})

`src/generated-routes.ts` is **committed** to the template тАФ it must exist for `tsc -b` (in the build script) to type-check `App.tsx`'s `import { routes } from './generated-routes'`. The file is regenerated by the plugin at `buildStart` (both `vite dev` and `vite build`), so subsequent diffs only appear when pages change. The file ships with `// @ts-nocheck` and `/* eslint-disable */` at the top, and ESLint additionally ignores it via `eslint.config.js`.

import { Moon, Sun } from 'lucide-react'
import { useTheme } from 'next-themes'

import { cn } from '~/lib/cn'

export function ThemeToggle({ className }: { className?: string }) {
  const { resolvedTheme, setTheme } = useTheme()
  const isDark = resolvedTheme === 'dark'
  return (
    <button
      type="button"
      aria-label="Toggle theme"
      onClick={() => setTheme(isDark ? 'light' : 'dark')}
      className={cn(
        'inline-flex size-9 items-center justify-center rounded-md',
        'border border-border bg-background text-text',
        'hover:bg-background-secondary transition-colors',
        className,
      )}
    >
      {isDark ? <Sun className="size-4" /> : <Moon className="size-4" />}
    </button>
  )
}

## Routing

Add a file under `src/pages/` and it becomes a route automatically.

- `pages/foo.tsx` тЖТ `/foo`
- `pages/blog/[id].tsx` тЖТ `/blog/:id`
- `pages/(group)/layout.tsx` тЖТ wraps grouped routes
- Add `.sync.tsx` to opt out of lazy loading

Each page file must export a named `Component` (and optional `loader`). See [vite-plugin-route-builder](https://www.npmjs.com/package/vite-plugin-route-builder).

## Theming

Light/dark is driven by `next-themes` via `data-theme="dark"`. Use semantic Pastel classes тАФ `bg-background`, `text-text`, `border-border`, `bg-accent` тАФ and they switch automatically. See the [Pastel docs](https://github.com/Innei/Pastel) for the full token list.

## State

`jotai` provider is wired in `providers.tsx`. Define atoms inside the relevant `features/<domain>/atoms.ts` and consume with `useAtom`.

## Editor jump

Hold **Alt** and click any DOM node in the browser тАФ the file opens in Cursor at the right line. Configure `editor` in `vite.config.ts` for other editors (`vscode`, `webstorm`, etc.).
````

## 9. Implementation order

1. Create `templates/react-starter/` directory tree.
2. Write `package.json` with deps from ┬з5.
3. Write tooling configs (┬з6): `vite.config.ts`, three tsconfigs, `eslint.config.js`, `prettier.config.mjs`, `.gitignore`.
4. Write entry/HTML (`index.html`, `main.tsx`, `env.d.ts`).
5. Write providers, App, styles.
6. Write `components/ui/button.tsx`, `lib/cn.ts`.
7. Write `features/theme/` (toggle + barrel).
8. Write pages: `layout.tsx`, `index.tsx`, `about.tsx`.
9. Run `pnpm i && pnpm dev` once inside the starter тАФ this triggers `vite-plugin-route-builder` to generate `src/generated-routes.ts`. Commit the generated file.
10. Run `pnpm build && pnpm lint && pnpm typecheck` inside the starter to verify it bootstraps cleanly.
11. Write `templates/react-starter/README.md`.
12. Append the Starter Templates section to the root `README.md`.
13. Commit.
